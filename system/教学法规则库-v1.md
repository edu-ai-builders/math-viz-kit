# 数学可视化教学法规则库 v1

本文档将学习科学文献结论转化为可执行的设计规则。  
每条规则包含：**一句话指令 → 研究依据 → 如何在 HTML 中落地 → 常见反面案例**。

所有数学可视化项目在生成 PRD 和编写代码时，必须标注应用了哪些规则（如 `<!-- PEDAGOGY: V3 分步揭示 -->`）。

---

## 规则 V1 · 动态跃迁

> **只做"静态讲不清、动态一秒懂"的知识点。**

### 研究依据
Schoenherr & Schukajlow (2024) 的 meta-analysis 指出，可视化对数学学习整体有效（effect size g ≈ 0.5–0.65），但效果高度依赖"是否存在静态表达无法有效传递的核心关系"。GeoGebra 相关文献（PMC meta-analysis, 2021）同样表明，动态拖拽和参数变化——而非"更漂亮的图"——才是可视化有效的根本原因。

### 判断标准（满足任一即值得做）
- 这个概念包含"**变与不变**"的动态关系（如：参数改变，形状变化，某性质保持不变）
- 学生的**常见误区**可以被动态过程直接纠正（如：让学生看到分母增大时分数条反而变短）
- 静态图需要**多张**才能说清楚的，动态图只需**一帧**即可

### 在 HTML 中如何落地
- 每个 demo 必须有一个**核心动态关系**，在注释中写清楚：  
  `<!-- 核心动态关系：k 增大 → 直线斜率变陡 → 视觉上直线向上旋转 -->`
- `requestAnimationFrame` 或 CSS transition 用于体现这个关系
- 避免把"动画"当目的本身——纯装饰性动画（加载特效、颜色闪烁）应删除

### 反面案例
✗ 用动画逐字显示公式（这不是"动态关系"，是演示文稿风格）  
✗ 把本可用一张图说清楚的内容强行做成滑杆（为交互而交互）

## 规则 V2 · 表征桥接

> **多表征必须联动，不能并排静置。一个操作触发，多个表征同步响应。**

### 研究依据
Frontiers in Psychology (2020) 的多表征研究明确指出：多表征学习有效的前提是帮助学习者建立**表征之间的映射关系**（representational connections），而不是"同屏堆叠更多内容"。先验知识较低的学习者尤其需要逐步引导式表征切换，过早呈现全部表征反而增加认知负荷。

### 表征类型参考
- **操作层**：拖拽、滑杆、输入
- **图形层**：坐标图、几何图形、数轴
- **数值层**：数字、表格、计算结果
- **符号层**：代数表达式、公式

### 在 HTML 中如何落地
```js
// 正确做法：一个事件触发所有表征同步更新
function onSliderChange(k) {
  updateGraph(k);      // 图形层同步
  updateTable(k);      // 数值层同步
  updateFormula(k);    // 符号层同步
  showBridgeFeedback(k); // 桥接说明更新
}
```
- 表征之间用**颜色编码**建立对应：图上的红色点 = 表格里的红色行 = 公式里的红色变量
- 禁止让两个表征同时静态存在而无交互连接

### 反面案例
✗ 左边一个函数图像，右边一个数值表，两者无任何同步联动  
✗ 切换"图像/表格"标签页（彼此独立）而不是同步显示

## 规则 V3 · 分步揭示

> **初始状态只显示最小可理解单元。用户主动触发，才展开下一层。永远提供重置入口。**

### 研究依据
Cognitive Load Theory（Sweller, 2010；Mayer & Moreno, 2003）将认知负荷分为三类：
- **Intrinsic load**（内在负荷）：知识本身的复杂度，无法消除
- **Germane load**（相关负荷）：对理解有贡献的认知努力，应保留
- **Extraneous load**（无关负荷）：设计不良造成的额外负担，应消除

一次性呈现所有信息是 extraneous load 的主要来源。

### 在 HTML 中如何落地
```js
// 三阶段揭示模式（适用于大多数项目）
const STAGES = {
  1: () => { showCore(); hideAll(['labels','formula','extras']); },
  2: () => { showCore(); showLabels(); hideAll(['formula','extras']); },
  3: () => { showAll(); }
};

// 每个阶段对应的讲解文字也分步更新
```
- 页面初始状态：只显示**核心图形/对象**，无标注、无公式
- 提供"下一步"或"点击展开"按钮，不要自动全部播放
- 必须有**重置按钮**，让学生可以重新从头探索
- 推荐在代码注释中标记：`<!-- 阶段 1：仅显示三角形 -->`

### 反面案例
✗ 打开页面就看到完整的"标注 + 公式 + 图 + 表 + 解释"，学生不知道看哪里  
✗ 动画自动播放一次后停止，学生没有控制权

## 规则 V4 · 信号聚焦

> **颜色、箭头、高亮只用于指向一个核心关系，不做装饰。**

### 研究依据
Signaling Principle（Mayer, 2009）：在复杂图示中，加入视觉信号（颜色、箭头、框线）可以引导学习者注意力，**但前提是信号只指向真正重要的关系**。信号过多（每个元素都有颜色、每条边都有标注）会导致信号失效，学习者无法分辨哪个才是重点。

### 在 HTML 中如何落地
- 每个页面最多使用 **3 种强调色**，每种颜色只对应一类信息
- 动画高亮（如描边闪烁）在同一时刻只作用于**一个**对象
- 箭头和标注：出现时机由交互触发，不常驻
- CSS 命名规范：`--color-primary`（主关系）、`--color-secondary`（辅助）、`--color-neutral`（背景）

```css
/* 示例：颜色只传达一个语义 */
:root {
  --color-subject: #2a5c8f;  /* 主角：直角三角形斜边 */
  --color-leg-a:   #0f6e56;  /* 直角边 a */
  --color-leg-b:   #854f0b;  /* 直角边 b */
  /* 不添加第4种强调色 */
}
```

### 反面案例
✗ 用7种不同颜色区分7条不同的线，学生需要先读图例  
✗ 鼠标悬浮任何地方都有高亮效果，导致学生注意力分散

## 规则 V5 · 操作先于符号

> **让学生先通过操作建立感知，再出现数学符号和公式。**

### 研究依据
具身认知研究（Abrahamson & Trninic, 2015；Tran et al., 2017 Cognitive Research）表明：学习者通过**感知运动行动**（perceptual-motor action）建立的直觉，为后续符号形式化提供了认知锚点。先给公式再给动画，会导致学生"认识公式但不理解公式"。

GeoGebra 研究脉络也一再验证：**探究先于讲解**（inquiry before instruction）在几何和函数概念学习中更有效。

### 在 HTML 中如何落地
```html
<!-- 正确顺序 -->
<!-- 阶段 1：让学生操作拖拽，感受三角形三边变化 -->
<!-- 阶段 2：标注边长数值，学生可以观察数字 -->
<!-- 阶段 3：显示 a² + b² = c² 的公式 -->

<!-- 错误顺序 -->
<!-- 先在页面顶部显示 a² + b² = c²，然后让学生验证 -->
```
- 公式/代数表达式默认隐藏，由用户主动点击"查看公式"才展开
- 或在用户完成操作后，公式以动画形式"由现象归纳出来"

### 反面案例
✗ 页面顶部大字显示公式，下方是"验证此公式"的互动区  
✗ 先定义术语，再让学生去"看"定义描述的现象

## 规则 V6 · 动作—数学一致

> **交互动作的方向和形式，要对应数学结构本身。**

### 研究依据
Gestural Congruency 原则（Alibali & Nathan, 2012；Frontiers in Education, 2020）：当手势/操作的运动模式与数学概念的内在结构一致时（gestural congruency），学习迁移效果显著优于不一致的设计。

例：函数增大 → 手指/滑杆向上移动 ✓  
函数增大 → 点击"+1"按钮 ✗（抽象，与数学方向无关联）

### 在 HTML 中如何落地

| 数学概念 | 对应的交互方向/形式 |
|---------|-----------------|
| 正方向移动（加正数） | 向右拖动 |
| 负方向移动（加负数） | 向左拖动 |
| 函数增大（k > 0） | 滑杆向右 → 图线变陡 |
| 角度增大 | 顺时针旋转弧线 |
| 放大 | 向外捏合/向右拖动"缩放"滑杆 |
| 分割/分解 | 实际拖拽图形使之分开 |
| 函数上移（b 增大） | 整条直线向上平移 |

- 在注释中标注：`<!-- V6：向右拖动 = 增大正数，与数轴方向一致 -->`

### 反面案例
✗ "斜率增大"但线条反而向下旋转  
✗ 拖拽角色向右走，数轴数值却向左增大  
✗ 用文本输入框输入角度值，而不是旋转操作

## 规则 V7 · 单一认知动作

> **一个 demo 只承载一个主要认知动作（比较/分解/变换/逼近/累积/缩放/投影/模拟）。**

### 研究依据
认知负荷理论中的"分割注意效应"（Split-attention Effect）：当一个页面同时要求学习者完成多个认知任务时，注意资源被分散，每个任务的学习效果均下降。数学可视化的"信息密度陷阱"很常见——在一个页面里同时想讲"函数性质"+"参数变化"+"解方程"+"图像几何意义"，结果每件事都没讲透。

### 在 HTML 中如何落地
- PRD 阶段必须确定唯一的 `cognitive_action`，并在代码注释顶部标注：
  ```js
  // 核心认知动作：变换（观察参数 k、b 如何独立控制直线的两种性质）
  // 不在此页面讲解：方程求解、函数应用题、斜率的物理意义
  ```
- `suitable_for_*` 标签也应体现聚焦：一个 demo 只"适合"1–2 个场景
- 如果一个知识点涉及多个认知动作，应**拆分为多个独立 demo**

### 反面案例
✗ 一次函数页面同时包含：参数调节 + 方程求解 + 函数应用题 + 图像变换  
✗ 在单位圆页面同时教：三角函数定义 + 诱导公式 + 正弦定理 + 极坐标

## 规则 V8 · 概念反馈

> **反馈不只说对错，要说"你刚才发现了什么"。**

### 研究依据
Formative feedback 研究（Black & Wiliam, 1998；Shute, 2008）区分了两种反馈：
- **结果反馈**（KOR, Knowledge of Result）：只告诉对错，学习效果弱
- **解释性反馈**（Elaborated Feedback）：解释原因、揭示规律，学习效果强，尤其对概念理解

在数学可视化场景中，最有价值的反馈是**揭示不变量**——"无论你怎么操作，这件事始终成立"。这是数学本质的核心表达。

### 概念反馈的三种模式

**模式一：不变量揭示**  
> "无论你怎么移动直角三角形上的点，a² + b² 始终等于 c²。这就是勾股定理。"

**模式二：规律归纳**  
> "你注意到了吗？当 k 为正数时，直线从左下到右上；k 为负数时，方向相反。"

**模式三：操作回顾**  
> "你刚才向右跳了 3 格，从 5 到达了 8，所以 5 + 3 = 8。"

### 在 HTML 中如何落地
```js
// 概念反馈区域（独立的 DOM 元素）
function showConceptFeedback(action, result) {
  const messages = {
    'move_right': `你向右跳了 ${result.steps} 格，所以 ${result.start} + ${result.steps} = ${result.end}。`,
    'k_positive': `斜率 k = ${result.k} 是正数，所以直线从左下方延伸到右上方。`,
    'pythagorean': `验证：${result.a}² + ${result.b}² = ${result.a2+result.b2} = ${result.c}² ✓`
  };
  document.getElementById('conceptFeedback').textContent = messages[action];
}
```
- 概念反馈区域固定在页面显眼位置（不是弹窗，不会消失）
- 文字用第二人称"你"，像老师在旁边说话
- 禁止只显示 ✓ 或 ✗ 而不附加任何说明

### 反面案例
✗ 只显示绿色对号/红色叉号  
✗ 弹出对话框说"回答正确！"然后关闭  
✗ 反馈文字太长（超过2句），学生不会读

## 快速查阅表

| 规则 | 核心问题 | 检查方式 |
|------|---------|---------|
| V1 动态跃迁 | 这个知识点有"变与不变"吗？ | 找到核心动态关系后才开始写代码 |
| V2 表征桥接 | 操作时，所有表征同步更新了吗？ | 触发一次事件，检查所有视图是否跟随变化 |
| V3 分步揭示 | 初始页面是否尽量简洁？ | 打开页面，数一数同时可见的信息层数 |
| V4 信号聚焦 | 颜色有没有超过3种强调色？ | 统计 CSS 变量中的强调色数量 |
| V5 操作先于符号 | 公式是在操作之前还是之后出现的？ | 查看初始渲染时公式是否默认隐藏 |
| V6 动作—数学一致 | 拖拽方向是否对应数学方向？ | 用手势描述操作，对应是否自然 |
| V7 单一认知动作 | 这个页面的认知动作是唯一的吗？ | 能否用一个动词（比较/变换…）概括全部交互 |
| V8 概念反馈 | 反馈是否揭示了数学规律？ | 找到 `showConceptFeedback` 的调用，读反馈文字 |
